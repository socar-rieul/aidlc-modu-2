# User Stories (테이블오더 서비스)

본 문서는 Story Generation Plan의 5문항 답변에 따라 작성된 스토리 모음이다.

- **Format**: Connextra (`As a [persona], I want [goal], so that [benefit]`)
- **Acceptance Criteria**: Given / When / Then (Gherkin)
- **Breakdown**: Feature-Based — `requirements.md`의 FR-C1~C5 / FR-A1~A4 단위 1:N 도출
- **Granularity**: Small — 한 화면/인터랙션 = 1 스토리 (원칙)
- **Persona Refs**: [P1] 고객 진우 / [P4] 고객 정희(디지털 리터러시 부족 50대) / [P2] 점주 수민 / [P3] 알바 도윤 — 정의는 [personas.md](personas.md)

총 **23개 스토리** (고객 10 + 관리자 13). 사용자 추가 요청으로 P4 페르소나 + 디지털 리터러시 보조 스토리 2개(US-C0.1, US-C0.2) 보강.

---

## 0. 공통 비즈니스 룰 (Cross-cutting Rules)

스토리 본문에서 반복 인용되는 공통 룰. 각 스토리의 AC에서 `[CR-#]` 로 참조.

- **CR-1 매장ID 스코프 격리**: 모든 조회·쓰기·집계는 인증된 매장ID(또는 테이블이 속한 매장ID) 범위로 제한된다. 타 매장 데이터는 어떤 경로로도 노출되지 않는다.
- **CR-2 테이블 세션 라이프사이클**: 세션 = 첫 주문 시작 ~ 매장 이용 완료 처리. 세션 시작이 없으면 첫 주문 시점에 자동 생성. 세션 종료 시 해당 세션의 모든 주문은 `OrderHistory`로 이동하고, 테이블 현재 주문·총액은 0으로 리셋된다.
- **CR-3 현재 세션 가시성**: 고객 화면(메뉴·주문 내역)은 **현재 세션 주문만** 노출. 종료된 세션 주문은 고객 측에서 보이지 않는다.
- **CR-4 주문 스냅샷 보존**: 주문 확정 시 메뉴명·단가를 스냅샷으로 저장. 이후 메뉴 가격·이름 변경이 과거 주문 금액·표기에 영향 주지 않는다.
- **CR-5 비밀번호 해싱**: 관리자 비밀번호 + 테이블 비밀번호는 bcrypt 해시로 저장. 평문 저장·로그·응답 모두 금지.

## 1. 고객용 스토리 (FR-C)

### US-C0.1 — 처음 사용자 도움말·튜토리얼 (관련 NFR-4 확장)

**Story**
> **As a** 디지털 리터러시 부족 50대 고객 정희 [P4],
> **I want** 처음 태블릿을 만졌을 때 짧고 친절한 사용법 안내가 나오고, 헷갈릴 때 다시 볼 수 있는 도움말 버튼이 있길 원한다,
> **so that** 직원을 부르거나 일행에게 의존하지 않고 혼자서도 주문을 끝낼 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 첫 진입 시 자동 안내
  Given 이 태블릿에서 한 번도 도움말 완료 처리를 한 적이 없을 때
  When 정희가 메뉴 화면에 진입하면
  Then 3~5단계 짧은 가이드(메뉴 선택→장바구니→확정→내역) 오버레이가 나타나고
    And 큰 글자·큰 "다음" 버튼으로 단계별 진행이 가능하며
    And 마지막 단계에서 "다시 보지 않기"를 선택하면 localStorage에 완료 플래그가 저장된다.

Scenario: 헬프 버튼 재호출
  Given 정희가 메뉴/장바구니/주문 내역 어느 화면에 있든
  When 화면 상단의 "도움말" 버튼을 누르면
  Then 현재 화면에 맞춘 짧은 안내(이미지 + 1~2줄)가 다시 노출되고
    And 닫기 버튼이 명확히 표시된다.

Scenario: 일행이 끄고 사용해도 안전
  Given P1 진우가 가이드를 닫고 빠르게 사용 중일 때
  When 다음 손님 정희가 같은 세션에 진입하면
  Then 가이드는 즉시 자동 표시되지 않지만 헬프 버튼은 항상 노출돼 있어 호출 가능하다.
```

### US-C0.2 — 큰 글자·고대비 모드 토글 (관련 NFR-4 확장)

**Story**
> **As a** 디지털 리터러시 부족 50대 고객 정희 [P4],
> **I want** 화면 어디서든 한 번의 동작으로 글자·버튼·이미지 크기를 키우고 대비를 강화할 수 있길 원한다,
> **so that** 안경 없이도 메뉴를 또렷이 보고 정확히 고를 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 큰 글자 모드 켜기
  Given 정희가 메뉴 화면에 있는 상태
  When 상단의 "글자 크게" 토글을 누르면
  Then 전체 폰트가 1.5배 이상 확대되고
    And 버튼·카드의 터치 영역도 비례 확대되며 (최소 60×60px)
    And 한 화면당 노출 카드 수가 줄어드는 대신 스크롤로 보완된다.

Scenario: 고대비 모드 켜기
  Given 정희가 "고대비" 토글을 누른 상태
  When 화면이 갱신되면
  Then 텍스트·배경 명도 대비가 WCAG AA 이상으로 강화되고
    And 핵심 버튼은 색뿐 아니라 외곽선·아이콘으로도 구분된다.

Scenario: 설정 유지
  Given 정희가 두 토글을 모두 켠 상태
  When 새로고침 또는 같은 테이블 재진입이 발생하면
  Then 설정은 localStorage로 유지되어 재적용된다.

Scenario: 다음 손님을 위한 리셋
  Given 관리자가 US-A3.3으로 세션을 종료할 때
  When 세션 종료 처리가 끝나면
  Then 도움말 완료 플래그·UI 토글 설정도 함께 리셋된다 (다음 손님은 기본 상태로 시작).
```

### US-C1.1 — 태블릿 자동 로그인 (관련 FR-C1)

**Story**
> **As a** 매장 고객 진우 [P1],
> **I want** 테이블 태블릿을 켜자마자 별도 입력 없이 메뉴 화면을 보고 싶다,
> **so that** 직원을 부르거나 로그인 절차를 거치지 않고 바로 주문을 시작할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 사전 설정된 태블릿 자동 인증 성공
  Given 관리자가 이 태블릿에서 US-A3.1으로 매장ID·테이블번호·테이블비번을 1회 설정한 상태이고
    And localStorage에 인증 토큰이 저장돼 있고
  When 진우가 태블릿 브라우저를 열면
  Then 추가 입력 없이 메뉴 기본 화면이 표시되고
    And 화면 상단에 매장명·테이블번호가 노출되며
    And [CR-1] 응답은 해당 매장의 데이터만 포함한다.

Scenario: localStorage 인증 정보 누락 시 안내
  Given localStorage에 인증 정보가 없거나 손상돼 있을 때
  When 진우가 태블릿을 열면
  Then "관리자를 호출해 주세요" 안내 화면이 표시되고
    And 메뉴 화면으로는 진입할 수 없다.
```

### US-C2.1 — 메뉴 탐색·상세 조회 (관련 FR-C2)

**Story**
> **As a** 매장 고객 진우 [P1],
> **I want** 카테고리별로 메뉴 목록을 빠르게 훑고 각 메뉴의 사진·가격·설명을 보고 싶다,
> **so that** 무엇을 시킬지 빠르게 결정할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 기본 진입 시 메뉴 화면 노출
  Given 자동 로그인이 완료된 상태
  When 진우가 앱에 진입하면
  Then 메뉴 화면이 기본 화면으로 표시되고
    And 카테고리 목록이 상단에 노출된다.

Scenario: 카테고리 전환
  Given 메뉴 화면이 표시된 상태
  When 진우가 "음료" 카테고리를 탭하면
  Then 해당 카테고리 메뉴만 카드 형태로 표시되고
    And 카드에는 메뉴명·가격·이미지가 포함된다.

Scenario: 메뉴 상세 보기
  Given 메뉴 카드가 노출된 상태
  When 진우가 특정 메뉴 카드를 탭하면
  Then 메뉴명·가격·설명·이미지가 포함된 상세 뷰가 열린다.

Scenario: 터치 영역 보장
  Given 메뉴 화면이 표시된 상태
  When 모든 탭 가능한 버튼·카드를 측정하면
  Then 각 영역은 최소 44×44px 이상이다.
```

### US-C3.1 — 장바구니 추가·수량 조정·삭제 (관련 FR-C3)

**Story**
> **As a** 매장 고객 진우 [P1],
> **I want** 메뉴를 장바구니에 담고 수량을 늘리거나 줄이고 삭제하고 싶다,
> **so that** 일행과 의논해서 최종 주문 구성을 자유롭게 조정할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 메뉴 장바구니 추가
  Given 메뉴 상세 뷰가 열린 상태
  When 진우가 "담기"를 탭하면
  Then 장바구니에 해당 메뉴가 1개 단위로 추가되고
    And 장바구니 아이콘에 항목 수가 증가한다.

Scenario: 수량 증가·감소
  Given 장바구니에 메뉴가 담긴 상태
  When 진우가 "+" 또는 "-" 버튼을 탭하면
  Then 해당 메뉴 수량이 변경되고
    And 수량이 0이 되면 장바구니에서 자동 제거된다.

Scenario: 단일 항목 삭제
  Given 장바구니에 다수 항목이 담긴 상태
  When 진우가 특정 항목의 "삭제"를 탭하면
  Then 해당 항목만 제거되고 나머지는 유지된다.
```

### US-C3.2 — 장바구니 총액 표시 + 비우기 (관련 FR-C3)

**Story**
> **As a** 매장 고객 진우 [P1],
> **I want** 장바구니 총액을 실시간으로 보고 한 번에 비울 수 있길 원한다,
> **so that** 예산 안에서 주문하고, 잘못 담은 경우 빠르게 처음부터 다시 시작할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 총액 실시간 계산
  Given 장바구니에 항목이 0개 이상 있는 상태
  When 항목 추가·수량 변경·삭제가 발생할 때마다
  Then 총액이 즉시(<200ms) 재계산되어 화면에 표시된다.

Scenario: 전체 비우기
  Given 장바구니에 1개 이상 항목이 있는 상태
  When 진우가 "장바구니 비우기"를 탭하고 확인 팝업에서 "예"를 누르면
  Then 장바구니가 빈 상태가 되고 총액은 0원으로 표시된다.

Scenario: 비우기 취소
  Given "장바구니 비우기" 확인 팝업이 열린 상태
  When 진우가 "아니오"를 누르면
  Then 장바구니는 그대로 유지된다.
```

### US-C3.3 — 새로고침 후 장바구니 유지 (관련 FR-C3, NFR-5)

**Story**
> **As a** 매장 고객 진우 [P1],
> **I want** 실수로 화면을 새로고침해도 장바구니에 담은 항목이 사라지지 않길 원한다,
> **so that** 처음부터 다시 메뉴를 고르는 수고를 하지 않아도 된다.

**Acceptance Criteria**

```gherkin
Scenario: 새로고침 후 복원
  Given 장바구니에 N개 항목·총액 X원이 표시된 상태
  When 진우가 브라우저를 새로고침하면
  Then 새로고침 후에도 동일한 N개 항목·총액 X원이 복원된다.

Scenario: 주문 확정 후 자동 비움
  Given 주문 확정이 성공한 직후
  When 진우가 새로고침하면
  Then 장바구니는 빈 상태로 유지된다.

Note: 장바구니는 서버에 전송되지 않고 클라이언트(localStorage 등)에만 저장된다.
```

### US-C4.1 — 주문 확정 성공 플로우 (관련 FR-C4)

**Story**
> **As a** 매장 고객 진우 [P1] (및 정희 [P4]),
> **I want** 장바구니의 메뉴를 한 번에 주문으로 확정하고, 주문번호를 받은 뒤 다음 메뉴를 이어서 고를 수 있길 원한다,
> **so that** 한 번 앉아서 여러 차례 추가 주문하는 흐름이 자연스럽고, 실수로 비싼 메뉴를 잘못 시키지도 않는다.

**Acceptance Criteria**

```gherkin
Scenario: 주문 확정 성공
  Given 장바구니에 1개 이상 항목이 있고 서버가 정상 응답할 때
  When 진우가 "주문 확정"을 탭하면
  Then 주문 페이로드(매장ID·테이블ID·세션ID·메뉴 목록[메뉴명·수량·단가]·총액)가 서버로 전송되고
    And [CR-4] 메뉴명·단가는 주문 시점 값으로 스냅샷 저장되며
    And [CR-2] 해당 테이블에 활성 세션이 없으면 신규 세션이 자동 생성된다.

Scenario: 성공 직후 화면 흐름
  Given 주문이 성공한 직후
  When 응답이 도착하면
  Then 주문번호가 5초간 표시되고
    And 장바구니는 자동으로 비워지고
    And 5초 후 메뉴 화면으로 자동 리다이렉트된다.

Scenario: 최종 확정 직전 강화 확인 (디지털 리터러시 보조)
  Given 정희가 "주문 확정" 버튼을 눌렀을 때
  When 서버 전송 직전에
  Then 메뉴/수량/총액을 큰 글자로 요약한 확인 다이얼로그가 표시되고
    And "주문하기"와 "취소" 버튼이 60×60px 이상으로 분리 배치되며
    And "취소"를 누르면 장바구니로 돌아가고 어떤 요청도 서버로 가지 않는다.
    And "주문하기"를 누른 경우에만 위의 "주문 확정 성공" 플로우가 진행된다.
```

### US-C4.2 — 주문 실패 처리 (관련 FR-C4)

**Story**
> **As a** 매장 고객 진우 [P1],
> **I want** 주문이 실패했을 때 명확한 에러 메시지를 보고 장바구니가 보존되길 원한다,
> **so that** 처음부터 다시 메뉴를 담지 않고 재시도할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 네트워크 오류
  Given 서버 통신이 실패하는 상태(타임아웃·5xx)
  When 진우가 "주문 확정"을 탭하면
  Then "주문에 실패했어요. 잠시 후 다시 시도해주세요." 메시지가 표시되고
    And 장바구니 항목·수량·총액은 그대로 유지된다.
```

```gherkin
  Then 해당 기간 주문만 표시된다.

Scenario: 닫기로 대시보드 복귀
```

### US-A4.1 — 메뉴 등록 (관련 FR-A4)

**Story**
> **As a** 점주 수민 [P2],
> **I want** 메뉴명·가격·설명·카테고리·이미지URL을 입력해서 새 메뉴를 추가하고 싶다,
> **so that** 시즌 신메뉴를 즉시 손님에게 노출할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 정상 등록
  Given 메뉴 관리 화면에서 모든 필수 필드를 입력했을 때
  When 수민이 "등록"을 누르면
  Then 신규 메뉴가 저장되고
    And 메뉴 목록에 즉시 반영되며
    And [CR-1] 본인 매장 메뉴 목록에만 노출된다.

Scenario: 필수 필드 누락
  Given 메뉴명·가격 중 하나라도 비어 있을 때
  When 수민이 "등록"을 누르면
  Then 누락 필드에 에러 표시가 나오고
    And 저장은 거부된다.

Scenario: 가격 범위 검증
  Given 가격에 음수 또는 0이 입력됐을 때
  When 수민이 "등록"을 누르면
  Then "가격은 1원 이상이어야 합니다" 에러가 표시된다.
```

### US-A4.2 — 메뉴 수정·삭제 (관련 FR-A4)

**Story**
> **As a** 점주 수민 [P2],
> **I want** 기존 메뉴의 가격·설명·이미지를 수정하거나 단종된 메뉴를 삭제하고 싶다,
> **so that** 메뉴판이 항상 최신 상태로 유지된다.

**Acceptance Criteria**

```gherkin
Scenario: 메뉴 수정 + 과거 주문 보호
  Given 기존 메뉴 "아메리카노"가 4,000원으로 등록돼 있고 과거 주문에 포함돼 있을 때
  When 수민이 가격을 4,500원으로 수정·저장하면
  Then 메뉴 카드 가격이 4,500원으로 갱신되고
    And [CR-4] 기존 OrderHistory의 아메리카노 금액은 4,000원으로 그대로 보존된다.

Scenario: 메뉴 삭제
  Given 메뉴 목록에 메뉴 N개가 있는 상태
  When 수민이 "삭제"를 누르고 확인 팝업에서 "예"를 선택하면
  Then 해당 메뉴가 목록에서 제거되고
    And [CR-4] 기존 주문 내 메뉴명·단가 스냅샷은 그대로 유지된다.

Scenario: 삭제 취소
  Given 삭제 확인 팝업이 열린 상태
  When 수민이 "아니오"를 누르면
  Then 메뉴는 그대로 유지된다.
```

### US-A4.3 — 메뉴 노출 순서 조정 (관련 FR-A4)
