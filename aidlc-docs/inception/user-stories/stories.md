## 0. 공통 비즈니스 룰 (Cross-cutting Rules) — v2 갱신

각 스토리 AC에서 `[CR-#]` 로 참조.

- **CR-1 매장ID 스코프 격리** (v1과 동일): 모든 조회·쓰기·집계는 인증된 매장ID(또는 테이블이 속한 매장ID) 범위로 제한. 타 매장 데이터는 어떤 경로로도 노출되지 않는다.
- **CR-2 테이블 세션 라이프사이클** (v2.2 변경): 세션 = **첫 QR 스캔 시점** ~ 매장 이용 완료 처리. 첫 스캔자가 활성 세션·공동 장바구니를 생성하고, 이후 스캔자는 **동일 활성 세션에 합류**(테이블당 활성 세션 1개). 참가자(`SessionParticipant`)는 QR 스캔으로 자유 합류·이탈하며 스캔 시점에 활성 세션으로 바인딩된다(sessionId non-null). 세션 종료 시 해당 세션의 모든 주문은 OrderHistory로 이동(단 **주문 0건 빈 세션은 OrderHistory 미기록**), 공동 장바구니는 비워지고, **모든 참가자 세션 토큰이 일괄 무효화**된다. 유휴 빈 세션 자동 만료는 없음(어드민 수동 종료만).
- **CR-3 현재 세션 가시성** (v2 재정의): 고객 화면(메뉴·장바구니·주문 내역)은 **현재 세션의 모든 참가자 주문을 합산해서** 표시. 참가자 식별 표기는 없음(MVP 단순화). 종료된 과거 세션 주문은 고객 측에 보이지 않는다.
- **CR-4 주문 스냅샷 보존** (v1과 동일): 주문 확정 시 메뉴명·단가를 스냅샷으로 저장. 이후 메뉴 가격·이름 변경이 과거 주문에 영향 주지 않는다.
- **CR-5 비밀번호·토큰 안전성** (v2 갱신): 관리자 비밀번호는 bcrypt 해시 저장(평문 금지). **QR 토큰은 UUIDv4 이상 엔트로피로 예측 불가**, 세션 토큰은 만료 시점에 무효화.
- **CR-6 공동 장바구니 동시성** (**신규**): 공동 장바구니는 **서버가 단일 권한**. 변경 이벤트마다 서버가 버전을 부여하고 SSE로 전 참가자에 푸시. 동시 편집은 **last-write-wins**, 클라이언트는 SSE 최신 상태로 즉시 갱신.
- **CR-7 광고 단방향** (**신규**): 광고는 시드 데이터로 등록된 `Advertisement` 슬롯에서만 노출. 외부 광고 서버 호출 없음, 매장별 분기·교체·통계 미수집. 광고주는 시스템 외부 stakeholder.

## 1. 고객용 스토리 (FR-C)

### US-C0.1 — 처음 사용자 도움말·튜토리얼 (관련 NFR-4 확장)

**Story**
> **As a** 디지털 리터러시 부족 50대 고객 정희 [P4],
> **I want** 본인 폰으로 처음 QR을 찍었을 때 짧고 친절한 사용법 안내가 나오고, 헷갈릴 때 다시 볼 수 있는 도움말 버튼이 있길 원한다,
> **so that** 직원이나 일행에게 매번 부탁하지 않고 혼자서도 주문을 끝낼 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: QR 스캔 후 첫 진입 시 자동 안내
  Given 본인 폰에서 이 세션에 한 번도 도움말 완료 처리를 한 적이 없을 때
  When 정희가 QR 스캔으로 메뉴 화면에 진입하면
  Then 3~5단계 짧은 가이드(메뉴 선택→공동 장바구니→확정→테이블 내역) 오버레이가 나타나고
    And 큰 글자·큰 "다음" 버튼으로 단계별 진행이 가능하며
    And 마지막 단계에서 "다시 보지 않기" 선택 시 본인 폰 localStorage에 완료 플래그가 저장된다.

Scenario: 헬프 버튼 재호출
  Given 정희가 어떤 화면에 있든
  When 화면 상단의 "도움말" 버튼을 누르면
  Then 현재 화면에 맞춘 짧은 안내(이미지 + 1~2줄)가 다시 노출되고 닫기 버튼이 명확하다.

Scenario: 세션 종료 시 리셋
  Given 관리자가 US-A3.3으로 세션을 종료할 때
  When 처리가 끝나면
  Then 모든 참가자 폰의 도움말 완료 플래그도 함께 리셋된다 (다음 손님은 기본 상태로 시작).
```

### US-C0.2 — 큰 글자·고대비 모드 토글 + 시스템 설정 따라가기 (관련 NFR-4)

**Story**
> **As a** 디지털 리터러시 부족 50대 고객 정희 [P4],
> **I want** 본인 폰의 시스템 폰트 크기 설정이 그대로 반영되고, 추가로 화면 내 토글로도 글자·대비를 키울 수 있길 원한다,
> **so that** 안경 없이도 메뉴를 또렷이 보고 정확히 고를 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 본인 폰 시스템 폰트 설정 반영
  Given 정희의 폰에 시스템 폰트 크기가 "크게"로 설정된 상태
  When 메뉴 화면이 로드되면
  Then 전 텍스트가 `rem` 단위로 시스템 설정에 비례 확대되고
    And 별도 토글 없이도 큰 글자로 노출된다.

Scenario: 사이트 내 큰 글자 모드 추가 토글
  Given 정희가 더 큰 글자를 원할 때
  When 상단의 "글자 더 크게" 토글을 누르면
  Then 전 폰트가 1.5배 추가 확대되고
    And 버튼·카드 터치 영역도 비례 확대된다 (최소 60×60px).

Scenario: 고대비 모드
  Given 정희가 "고대비" 토글을 누른 상태
  When 화면이 갱신되면
  Then 텍스트·배경 대비가 WCAG AA 이상이고 핵심 버튼은 색뿐 아니라 외곽선·아이콘으로도 구분된다.

Scenario: 본인 폰 단위 설정 유지
  Given 정희가 토글을 켠 상태
  When 새로고침 또는 같은 QR 재진입이 발생하면
  Then localStorage로 본인 폰에 유지되어 재적용된다 (다른 참가자 폰엔 영향 없음).
```

### US-C1.1 — QR 스캔 입장 (관련 FR-C1)

**Story**
> **As a** 매장 고객 진우 [P1] (또는 정희 [P4]),
> **I want** 테이블에 부착된 QR을 본인 폰 카메라로 찍으면 별도 가입·앱 설치 없이 바로 메뉴 화면으로 들어가고 싶다,
> **so that** 직원을 부르거나 절차를 거치지 않고 즉시 주문을 시작할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 정상 QR 스캔 입장
  Given 테이블에 영구 QR이 부착돼 있고 매장이 영업 중일 때
  When 진우가 본인 폰 카메라로 QR을 인식하고 링크를 열면
  Then [CR-5] 서버가 QR 토큰을 검증해 매장ID·테이블ID를 식별하고
    And [CR-2] 테이블에 활성 세션이 없으면 새 세션·공동 장바구니를 생성하고, 있으면 그 활성 세션에 합류시키며
    And 본인 폰에 새 `SessionParticipant` 토큰을 발급(활성 세션에 바인딩)해 localStorage에 임시 저장하며
    And 메뉴 기본 화면으로 자동 진입한다
    And 상단에 매장명·테이블번호가 노출된다.

Scenario: 같은 폰 재진입(idempotent)
  Given 진우 폰 localStorage에 유효한 세션 토큰이 이미 있을 때
  When 같은 QR 링크를 다시 열거나 새로고침하면
  Then 새 `SessionParticipant`를 중복 발급하지 않고 기존 토큰·세션을 그대로 재사용한다.

Scenario: 무효화된 QR
  Given 관리자가 운영상 사유로 해당 QR을 무효화한 상태
  When 진우가 QR을 스캔하면
  Then "잠시 후 다시 시도하거나 직원을 호출해주세요" 안내 화면이 표시되고 진입은 거부된다.

Scenario: 영업 종료/세션 없는 시각
  Given 매장이 영업 종료 상태일 때
  When 진우가 QR을 스캔하면
  Then 영업 종료 안내가 표시되고 진입은 거부된다.
```

### US-C2.1 — 메뉴 탐색·상세 조회 (모바일 반응형) (관련 FR-C2, NFR-11)

**Story**
> **As a** 매장 고객 진우 [P1] (및 정희 [P4]),
> **I want** 본인 폰 화면 폭에 맞게 카테고리·메뉴·상세를 쾌적하게 보고 싶다,
> **so that** 작은 화면에서도 무엇을 시킬지 빠르게 결정할 수 있다.

**Acceptance Criteria**

```gherkin
Scenario: 첫 진입 시 메뉴 화면 노출
  Given QR 스캔 입장이 완료된 상태
  When 진우가 앱에 진입하면
  Then 메뉴 화면이 기본 화면으로 표시되고
    And 카테고리 목록이 상단에 가로 스크롤 가능한 칩 형태로 노출된다.

Scenario: 카테고리 전환
  Given 메뉴 화면이 표시된 상태
  When 진우가 "음료" 칩을 탭하면
  Then 해당 카테고리 메뉴만 카드 형태로 표시되고 메뉴명·가격·이미지가 포함된다.

Scenario: 메뉴 상세 보기
  Given 메뉴 카드가 노출된 상태
  When 진우가 특정 메뉴 카드를 탭하면
  Then 메뉴명·가격·설명·이미지가 포함된 상세 뷰가 모달 또는 풀스크린으로 열린다.

Scenario: 반응형·터치 영역
  Given 화면 폭 320px ~ 480px 사이일 때
  When 모든 탭 가능한 버튼·카드를 측정하면
  Then 각 영역은 최소 44×44px 이상이고
    And 가로 스크롤 없이 세로 스크롤만으로 모든 콘텐츠 접근 가능하다.

Scenario: 품절 메뉴 표기
  Given 어떤 메뉴의 `soldout` 플래그가 true일 때
  When 메뉴 목록 또는 상세 화면이 렌더링되면
  Then 해당 메뉴 카드는 회색 처리되고 "품절" 배지가 표시되며
    And "담기" 버튼이 비활성화된다.
```

### US-C3.1 — 공동 장바구니 메뉴 추가·수량·삭제 (관련 FR-C3, CR-6)

**Story**
> **As a** 매장 고객 진우 [P1] (또는 일행 누구든),
> **I want** 본인 폰에서 공동 장바구니에 메뉴를 담고, 수량을 조정하고, 삭제할 수 있고, 그 변경이 일행 폰에 즉시 보이길 원한다,
> **so that** 누가 담든 일행 모두가 같은 장바구니를 본다.

**Acceptance Criteria**

```gherkin
Scenario: 본인 폰에서 추가
  Given 진우가 메뉴 상세 뷰가 열린 상태
  When "담기"를 탭하면
  Then 서버에 추가 요청이 전송되고
    And 서버가 새 장바구니 버전을 발행하며
    And [CR-6] SSE로 전 참가자에 변경 이벤트가 푸시된다.

Scenario: 일행 폰에 ≤ 2초 반영
  Given 정희가 같은 테이블 세션 다른 참가자로 합류해 있을 때
  When 진우가 메뉴를 담은 직후
  Then 2초 이내 정희 폰의 장바구니 아이콘·항목 수가 자동 갱신된다.

Scenario: 수량 증감 / 삭제
  Given 장바구니에 항목이 있는 상태
  When 누군가가 "+", "-", "삭제"를 탭하면
  Then 서버가 변경을 처리하고 SSE로 전 참가자에 푸시하며
    And 수량이 0이 되면 자동 제거된다.

Scenario: 품절 메뉴 담기 차단
  Given 메뉴의 `soldout`이 true인 상태
  When 누군가 해당 메뉴 카드의 비활성화된 "담기"를 시도하거나 API로 직접 추가 요청하면
  Then 클라이언트는 차단하고 서버도 4xx로 거부하며 장바구니는 변하지 않는다.

Scenario: 담은 후 품절 전환 시
  Given 장바구니에 메뉴가 담긴 상태에서 점주가 그 메뉴를 품절 처리할 때
  When 다음 SSE 동기화 또는 주문 확정 시점에
  Then 장바구니의 해당 항목 옆에 "품절" 표시가 나타나고 다른 참가자에게도 동시 안내되며
    And 주문 확정 시점에는 서버가 거부해 "품절 메뉴가 포함되어 있어요. 빼고 다시 시도해주세요" 안내를 반환한다.
```

### US-C3.2 — 공동 장바구니 비우기 + 총액 표시 (관련 FR-C3)

**Story**
> **As a** 매장 고객 진우 [P1] (및 정희 [P4]),
> **I want** 공동 장바구니 총액을 실시간으로 확인하고, 필요할 때 한 번에 비울 수 있길 원한다,
