## Q1 — 장바구니 모델 (BYOD에서 가장 핵심 결정)

여러 손님이 각자 폰으로 같은 테이블에 합류했을 때 장바구니를 어떻게 관리할까요?

A) **개인 장바구니** — 각 손님이 본인 폰에 자기 장바구니를 가지고, 본인 것만 주문 확정. 일행은 서로의 장바구니를 볼 수 없음 (가장 단순)

B) **공동 장바구니** — 테이블 세션 단위로 한 개의 장바구니. 일행 누구나 추가·수정 가능. 한 사람이 확정 시 전체 주문 (협업적)

C) **하이브리드** — 기본은 개인이지만 "테이블에 올리기" 버튼으로 공동 장바구니로 이동 가능

X) Other

[Answer]: B — 공동 장바구니 (테이블 세션 단위 1개)

## Q2 — 주문 가시성 (CR-3 "현재 세션 가시성" 재정의)

각 손님이 "주문 내역"을 열었을 때 무엇이 보일까요?

A) **본인 주문만** — 내가 직접 확정한 주문만 시간 역순 표시 (프라이버시 최우선)

B) **테이블 전체** — 같은 테이블 세션의 모든 참가자 주문을 합쳐서 표시 (계산서 느낌)

C) **둘 다 탭으로 분리** — "내 주문" 탭 + "테이블 전체" 탭

X) Other

[Answer]: B — 테이블 전체 (계산서 느낌)

## Q3 — QR 코드 라이프사이클 (인증·세션 모델)

테이블별 QR 코드는 언제 새로 발급/갱신될까요?

A) **세션마다 갱신** — 매장 이용 완료(US-A3.3) 시 QR이 무효화되고 새 QR 발급. 다음 손님은 새 QR 스캔 (보안·격리 최선)

B) **일정 시간마다 갱신** — 예: 24시간마다 자동 롤링. 운영 부담 적음, 노출 위험 약간 증가

C) **테이블당 영구 고정** — 한 번 인쇄한 QR을 계속 사용. 가장 단순하지만 외부 노출 시 무효화 어려움

X) Other

[Answer]: C — 테이블당 영구 고정 (한 번 인쇄, 외부 노출 시 무효화는 운영 영역)

## Q4 — BYOD 지원 범위 / 클라이언트 형태

손님이 QR을 스캔한 뒤 무엇으로 주문하게 할까요?

A) **모바일 웹만 (PWA)** — 기본 브라우저(iOS Safari / Android Chrome)에서 접속. 설치 불필요, 홈 화면 추가 옵션 (가장 가벼움 — 권장)

B) **모바일 웹 + 네이티브 앱 옵션** — 웹은 기본, 앱은 단골/매장 별 선택. 앱은 푸시 알림 가능 (스코프 큼)

C) **네이티브 앱 강제** — 무조건 앱 설치. UX 정교하나 진입장벽 높음

X) Other

[Answer]: A — 모바일 웹만 (PWA, 설치 불필요)

## Q5 — 광고 슬롯 포함 여부 (이전 대화 컨텍스트 반영)

사용자가 별도 메시지에서 "광고 영역(모두의주차장 광고)" 추가를 언급. v2 iteration에 함께 다룰까요?

A) **이번에 함께 포함** — 단방향 광고 슬롯(모두의주차장 배너) 1~2개를 메뉴/장바구니 화면에 노출. 광고주는 외부 stakeholder(시스템에 진입하지 않음). 광고 페르소나 추가 X (가장 단순 — 권장)

B) **광고 슬롯 포함 + 매장이 on/off·교체** — 점주가 매장별로 광고를 켜거나 다른 광고로 교체 가능 (스코프 중간)

C) **풀 광고 시스템** — 광고주 등록·노출·통계·정산 UI 포함. 광고주 페르소나·스토리 필요 (PoC 스코프 초과)

D) **이번엔 제외** — v2 BYOD 전환만 끝내고 광고는 별도 iteration에서 다룸

X) Other

[Answer]: A — 이번에 함께 포함 (단방향 모두의주차장 배너 1~2개, 광고주 페르소나 X)
